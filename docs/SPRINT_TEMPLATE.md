# Sprint Template & Delivery Standard

Every FactoryOS sprint follows the same structure and produces the same artifacts. This
standard implements the [Constitution](CONSTITUTION.md) — it does not replace or override it.

## 1. Sprint document sections

Each sprint is specified and executed against these sections, in order:

| Section | Purpose |
|---|---|
| **Objective** | The single outcome the sprint delivers. |
| **Scope** | What is built in this sprint. |
| **Out of Scope** | What is explicitly deferred, to prevent scope creep. |
| **Requirements** | Functional and non-functional requirements. |
| **Architecture** | How the work fits the layered, event-driven, plugin architecture. |
| **Folder** | The exact folder/project layout the sprint adds or changes. |
| **Interfaces** | The contracts (interfaces, events, manifests) introduced. |
| **Implementation** | Production-grade implementation of the interfaces. |
| **Validation** | How correctness is checked beyond compilation. |
| **Tests** | Unit, integration and architecture tests. |
| **Acceptance** | Objective pass/fail criteria for the sprint. |
| **Definition of Done** | The Constitution's DoD, applied to this sprint. |
| **Deliverables** | The concrete outputs listed in §2. |

## 2. Mandatory artifacts produced at the end of every sprint

No sprint is complete until all of the following are produced and green:

| Artifact | How it is produced |
|---|---|
| **Build** | `dotnet build FactoryOS.slnx -c Release` — 0 warnings, 0 errors. |
| **Unit Test** | `dotnet test` over `FactoryOS.Tests` — all pass. |
| **Integration Test** | `dotnet test` over `FactoryOS.IntegrationTests` against the `docker-compose` dependencies — all pass. |
| **Architecture Test** | `dotnet test` over `FactoryOS.ArchitectureTests` — layer, dependency and plugin rules enforced. |
| **Coverage Report** | Collected via Coverlet, rendered with ReportGenerator, stored under `artifacts/coverage/`. |
| **Performance Report** | BenchmarkDotNet results for performance-sensitive components, stored under `artifacts/performance/`. |
| **Security Report** | `dotnet list package --vulnerable`, analyzer results and secret scan, stored under `artifacts/security/`. |
| **Release Note** | Sprint summary under `docs/releases/`. |
| **Change Log** | Appended to [`CHANGELOG.md`](../CHANGELOG.md). |

## 3. Absolute rules (from the Constitution)

- No `TODO`, no mock, no fake service, no non-compiling code.
- The solution builds and every present test passes at the end of the sprint.
- Reports are generated by real tooling — they are never hand-authored or fabricated.

*Where a sprint's own text specifies fewer artifacts (e.g. an early skeleton sprint that only
requires a build), that sprint's text governs its scope; the full artifact set applies as soon
as the corresponding test/tooling projects exist in the roadmap.*
